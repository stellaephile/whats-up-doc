# Requirements Document

## Introduction

What's Up Doc is an AI-powered healthcare navigation platform for India that helps patients find the right level of care based on their symptoms. The system uses a two-stage AI assessment (Haiku for triage, Sonnet for full assessment) to evaluate symptom severity and route patients to appropriate healthcare facilities (PHC/Clinic/Hospital/Emergency) without providing medical diagnosis.

The platform addresses critical healthcare navigation problems in India:
- Patients rushing to expensive private hospitals for minor issues
- Visiting facilities lacking required specialization
- Emergency response delays costing lives

The system prioritizes safety through conservative emergency detection, government facility preference for mild/moderate cases, and progressive radius expansion to ensure patients always find appropriate care.

## Glossary

- **System**: The What's Up Doc healthcare navigation platform
- **Patient/User**: A person seeking healthcare guidance through the platform (anonymous, no login required)
- **Care_Level**: The appropriate level of healthcare (PHC, Clinic, Hospital, Emergency)
- **Severity_Level**: Classification of symptom urgency (mild/moderate/high/emergency)
- **Severity_Score**: Numerical score (1-10) indicating urgency of medical attention
- **Emergency_Symptoms**: Life-threatening symptoms requiring immediate medical attention
- **Facility**: A healthcare provider (PHC, clinic, hospital, emergency center)
- **Assessment**: Two-stage AI evaluation of symptoms (Stage 1: Haiku triage, Stage 2: Sonnet full assessment)
- **Routing**: Matching patients to appropriate healthcare facilities based on severity and location
- **Clarifying_Questions**: Follow-up questions generated by AI to improve assessment accuracy
- **Stage1_Cache**: Temporary storage of initial triage results for second-stage assessment
- **PHC**: Primary Health Centre - government facility for basic healthcare
- **CGHS**: Central Government Health Scheme
- **AYUSH**: Traditional medicine systems (Ayurveda, Yoga, Unani, Siddha, Homeopathy)
- **PostGIS**: PostgreSQL extension for geospatial queries
- **Progressive_Radius_Expansion**: Automatic search radius increase (3km → 5km → 10km → 20km) when no facilities found
- **Bedrock**: AWS service providing Claude AI models (Haiku and Sonnet)

## Functional Requirements

### Requirement 1: Intelligent Symptom Assessment

**User Story:** As a patient experiencing health symptoms, I want to describe my symptoms in natural language, so that the system can assess the severity and guide me to appropriate care.

#### Acceptance Criteria

1. WHEN a patient inputs symptoms in natural language (Hindi/English/Hinglish), THE System SHALL process the input and return initial triage within 3 seconds
2. WHEN symptoms are vague or incomplete, THE System SHALL generate clarifying questions (maximum 2 questions) in the same language as input
3. WHEN emergency keywords are detected in symptom input, THE System SHALL automatically upgrade the severity assessment to emergency level and skip clarifying questions
4. WHEN the assessment is complete, THE System SHALL assign a severity score (1-10) and severity level (mild/moderate/high/emergency) without providing medical diagnosis
5. THE System SHALL display appropriate disclaimers stating "Not a medical diagnosis. Consult a healthcare provider."

### Requirement 2: Emergency Detection and Response

**User Story:** As a patient experiencing emergency symptoms, I want the system to immediately identify life-threatening conditions, so that I can be directed to emergency care without delay.

#### Acceptance Criteria

1. WHEN emergency keywords are detected (chest pain, severe bleeding, breathlessness, stroke signs, loss of consciousness), THE System SHALL flag the case as emergency within 3 seconds
2. WHEN an emergency is detected, THE System SHALL display prominent red warning banner stating "These symptoms may be life-threatening"
3. WHEN emergency mode is activated, THE System SHALL skip clarifying questions and proceed directly to facility routing
4. WHEN in emergency mode, THE System SHALL display ambulance contact numbers (108/102) prominently in the warning banner
5. WHEN emergency is detected, THE System SHALL route only to facilities with emergency_available = TRUE

### Requirement 3: Healthcare Facility Routing

**User Story:** As a patient needing medical care, I want to be directed to appropriate healthcare facilities based on my symptom severity and location, so that I receive suitable care without unnecessary cost or delay.

#### Acceptance Criteria

1. WHEN symptoms indicate mild severity (score ≤3), THE System SHALL search for PHC, dispensaries, health centres, and clinics within initial radius of 5km
2. WHEN symptoms indicate moderate severity (score 4-6), THE System SHALL search for clinics, nursing homes, and hospitals within initial radius of 8km
3. WHEN symptoms indicate high severity (score 7-8), THE System SHALL search for hospitals and medical colleges within initial radius of 12km
4. WHEN emergency symptoms are detected (score 9-10), THE System SHALL search only for emergency-capable facilities within initial radius of 12km
5. WHEN no appropriate facilities are found within the initial radius, THE System SHALL progressively expand search radius (3km → 5km → 10km → 20km → 30km → 50km) and display expansion message to user
6. THE System SHALL use two-pass search strategy: Pass 1 (strict) filters by care type + specialty + distance, Pass 2 (relaxed) filters by care type + distance only

### Requirement 4: Specialty Department Matching

**User Story:** As a patient with specific symptoms requiring specialized care, I want to be directed to facilities with relevant departments, so that I reach the right specialist quickly.

#### Acceptance Criteria

1. WHEN pregnancy-related symptoms are detected, THE System SHALL prioritize facilities with Obstetrics & Gynecology in specialties_array
2. WHEN chest pain symptoms are detected (non-emergency), THE System SHALL prioritize facilities with Cardiology in specialties_array
3. WHEN respiratory symptoms are detected, THE System SHALL prioritize facilities with Pulmonology or ENT in specialties_array
4. WHEN neurological symptoms are detected, THE System SHALL prioritize facilities with Neurology in specialties_array
5. WHEN specialty filtering returns no results in strict mode, THE System SHALL fall back to relaxed mode (ignore specialty filter) to ensure facilities are found

### Requirement 5: Government Scheme Integration

**User Story:** As a cost-conscious patient, I want to be shown government facilities first for non-emergency cases, so that I can access affordable healthcare.

#### Acceptance Criteria

1. WHEN severity level is mild or moderate, THE System SHALL rank government facilities (hospital_category containing 'gov' or 'public') higher than private facilities in search results
2. WHEN displaying facility results, THE System SHALL show facility category (Government/Private/AYUSH) on each facility card
3. WHEN user applies "Sarkari" filter, THE System SHALL show only government facilities
4. WHEN user applies "Private" filter, THE System SHALL show only private facilities (non-government, non-AYUSH)
5. WHEN user applies "AYUSH" filter, THE System SHALL show only AYUSH facilities (ayush = TRUE or discipline_clean contains AYUSH disciplines)

### Requirement 6: Multi-Language Support

**User Story:** As a Hindi-speaking patient, I want to interact with the system in my preferred language, so that I can describe symptoms comfortably and understand the guidance.

#### Acceptance Criteria

1. THE System SHALL accept symptom input in Hindi, English, and Hinglish (mixed Hindi-English)
2. WHEN language is detected from user input, THE System SHALL generate clarifying questions in the same language
3. WHEN emergency keywords are detected in any supported language, THE System SHALL trigger emergency mode
4. THE System SHALL support emergency keywords in multiple languages (e.g., "chest pain", "seena dard", "dil ka daura")
5. THE System SHALL maintain language consistency throughout the clarifying questions chat flow

### Requirement 7: Location-Based Services

**User Story:** As a patient seeking nearby healthcare, I want the system to use my pincode to find the closest appropriate facilities, so that I can minimize travel time during illness.

#### Acceptance Criteria

1. WHEN a patient provides 6-digit pincode, THE System SHALL geocode the location using three-tier fallback strategy: (1) AWS Location Service, (2) Database exact pincode average, (3) District centroid
2. THE System SHALL calculate distances to healthcare facilities using PostGIS ST_Distance with accuracy of ±100 meters
3. WHEN displaying results, THE System SHALL show distance in kilometers with 1 decimal precision for each facility
4. THE System SHALL provide interactive Leaflet map visualization with color-coded facility markers based on severity level
5. THE System SHALL display radius circle on map showing the search area used

### Requirement 8: Data Privacy and Compliance

**User Story:** As a patient sharing sensitive health information, I want my data to be protected, so that my medical privacy is maintained.

#### Acceptance Criteria

1. THE System SHALL operate in anonymous mode without requiring user registration or login
2. THE System SHALL not persist symptom data in the database (assessment happens in-memory only)
3. THE System SHALL use HTTPS for all API communication to encrypt data in transit
4. THE System SHALL implement CORS restrictions to allow requests only from authorized frontend domains
5. THE System SHALL sanitize all user inputs to prevent injection attacks

### Requirement 9: System Performance and Reliability

**User Story:** As a patient in a medical emergency, I want the system to respond quickly and be available 24/7, so that I can get help when I need it most.

#### Acceptance Criteria

1. THE System SHALL complete Stage 1 assessment (Haiku triage) within 3 seconds under normal load
2. THE System SHALL complete Stage 2 assessment (Sonnet full assessment) within 5 seconds under normal load
3. THE System SHALL complete pincode geocoding within 1 second
4. THE System SHALL complete facility search query within 2 seconds
5. WHEN AI services are unavailable for emergency assessment, THE System SHALL retry maximum 2 times with exponential backoff (500ms base delay, ±20% jitter) before showing emergency facilities fallback
6. WHEN AI services are unavailable for non-emergency assessment, THE System SHALL retry maximum 4 times with exponential backoff (1000ms base delay, ±20% jitter) before showing all nearby facilities fallback
7. THE System SHALL display retry progress to user (e.g., "Assessing symptoms... (Attempt 2 of 4)")
8. THE System SHALL use jitter (±20% random variance) in retry delays to prevent thundering herd problem
9. WHEN all retry attempts fail, THE System SHALL display appropriate fallback results with warning message indicating AI service unavailability

### Requirement 10: Medical Disclaimer and Safety

**User Story:** As a healthcare platform, I want to ensure users understand the system's limitations, so that they seek appropriate medical care and don't rely solely on automated assessment.

#### Acceptance Criteria

1. THE System SHALL display "Not a medical diagnosis" disclaimer on all assessment results
2. THE System SHALL never state "You have [disease]" but only provide severity assessment and facility recommendations
3. WHEN providing facility recommendations, THE System SHALL include disclaimer "Consult healthcare provider for medical advice"
4. THE System SHALL display emergency contact numbers (108/102) prominently when emergency is detected
5. THE System SHALL use conservative emergency detection to prioritize patient safety (false positives preferred over false negatives)

### Requirement 11: Interactive Map Visualization

**User Story:** As a patient viewing facility results, I want to see facilities on an interactive map, so that I can visualize their locations and distances.

#### Acceptance Criteria

1. THE System SHALL display an interactive Leaflet map with OpenStreetMap tiles
2. THE System SHALL show color-coded markers for each facility based on severity level (green=mild, orange=moderate, dark orange=high, red=emergency)
3. THE System SHALL display a radius circle on the map showing the search area used
4. THE System SHALL auto-zoom the map to fit all facility markers and the radius circle
5. WHEN user clicks a facility marker, THE System SHALL display a popup with facility name, type, address, phone, and distance

### Requirement 12: Clarifying Questions Flow

**User Story:** As a patient with vague symptoms, I want the system to ask me clarifying questions, so that I can get more accurate facility recommendations.

#### Acceptance Criteria

1. WHEN Stage 1 assessment determines clarification is needed, THE System SHALL display a WhatsApp-style chat interface
2. THE System SHALL ask maximum 2 clarifying questions to avoid user fatigue
3. THE System SHALL display questions one at a time in a conversational flow
4. THE System SHALL show typing indicator before displaying each question
5. WHEN user answers all questions, THE System SHALL proceed to Stage 2 assessment with the provided answers
6. THE System SHALL provide a "Skip all questions" option for users who want to proceed immediately

### Requirement 13: Progressive Radius Expansion

**User Story:** As a patient in an area with few facilities, I want the system to automatically expand the search radius, so that I can find available facilities even if they're farther away.

#### Acceptance Criteria

1. WHEN no facilities are found within the initial radius, THE System SHALL automatically expand the search radius
2. THE System SHALL use expansion steps: 3km → 5km → 10km → 20km → 30km → 50km
3. WHEN radius is expanded, THE System SHALL display a message informing the user (e.g., "Expanded search to 10km to find suitable facilities")
4. WHEN no facilities are found even at maximum radius (50km), THE System SHALL display "No facilities found" message with emergency contact numbers
5. THE System SHALL try both strict mode (with specialty filter) and relaxed mode (without specialty filter) at each radius before expanding

### Requirement 14: Facility Filtering

**User Story:** As a patient with preferences for facility type, I want to filter results by Government/Private/AYUSH, so that I can see only the facilities I'm interested in.

#### Acceptance Criteria

1. THE System SHALL provide three filter toggles: "Sarkari" (Government), "Private", and "AYUSH"
2. WHEN "Sarkari" filter is disabled, THE System SHALL hide all government facilities from results
3. WHEN "Private" filter is disabled, THE System SHALL hide all private facilities from results
4. WHEN "AYUSH" filter is disabled, THE System SHALL hide all AYUSH facilities from results
5. THE System SHALL apply filters to both the facility list and map markers in real-time

### Requirement 15: Emergency Override Mode

**User Story:** As a patient in an emergency situation, I want to manually activate emergency mode, so that I can immediately see emergency facilities without going through assessment.

#### Acceptance Criteria

1. THE System SHALL provide an "Emergency SOS" button on the landing page
2. WHEN user clicks "Emergency SOS", THE System SHALL immediately set severity to emergency level
3. WHEN emergency override is activated, THE System SHALL skip symptom assessment and clarifying questions
4. WHEN emergency override is activated, THE System SHALL display emergency warning banner with ambulance numbers
5. WHEN emergency override is activated, THE System SHALL route only to emergency-capable facilities

## Correctness Properties

*Properties are universal characteristics that should hold true across all valid executions of the system.*

### Property 1: Symptom Processing Performance
*For any* symptom input in any supported language, the system SHALL process Stage 1 assessment and return results within 3 seconds under normal load conditions.

### Property 2: Emergency Detection and Response
*For any* symptom input containing emergency keywords (chest pain, severe bleeding, breathlessness, stroke signs, loss of consciousness), the system SHALL automatically flag as emergency, display warning message, skip clarifying questions, show ambulance numbers, and route only to emergency facilities.

### Property 3: Severity-Based Facility Routing
*For any* completed symptom assessment, the system SHALL route to appropriate facility types within correct initial radius: mild (≤3) → 5km, moderate (4-6) → 8km, high (7-8) → 12km, emergency (9-10) → 12km emergency-only.

### Property 4: Progressive Radius Expansion
*For any* facility search that returns zero results, the system SHALL automatically expand the search radius following the sequence: 3km → 5km → 10km → 20km → 30km → 50km.

### Property 5: Government Facility Prioritization
*For any* mild or moderate severity assessment, when government facilities are available, the system SHALL rank them higher than private facilities in search results.

### Property 6: Multi-Language Consistency
*For any* user session, when language is detected from initial input, the system SHALL maintain that language for clarifying questions throughout the session.

### Property 7: Location Services Accuracy
*For any* valid 6-digit pincode input, the system SHALL geocode within 1 second and calculate facility distances with ±100 meter accuracy using PostGIS.

### Property 8: Medical Disclaimer Enforcement
*For any* assessment result or facility recommendation, the system SHALL display appropriate disclaimers and never state diagnostic conclusions.

### Property 9: Emergency Override Behavior
*For any* user who activates emergency override mode, the system SHALL skip assessment, display emergency warning, and route only to emergency facilities.

### Property 10: Filter Application Consistency
*For any* facility filter toggle (Sarkari/Private/AYUSH), the system SHALL apply the filter to both the facility list and map markers simultaneously.

### Property 11: Retry Strategy with Jitter
*For any* AI assessment request that fails, the system SHALL retry with exponential backoff and jitter: emergency cases retry max 2 times (500ms base, ±20% jitter), non-emergency cases retry max 4 times (1000ms base, ±20% jitter), before falling back to showing nearby facilities.

## Non-Functional Requirements

### Performance
- Stage 1 assessment: < 3 seconds
- Stage 2 assessment: < 5 seconds
- Pincode geocoding: < 1 second
- Facility search: < 2 seconds
- Map rendering: < 1 second

### Scalability
- Support 1,000 concurrent users
- Handle 10,000 assessments per day
- Database query optimization with PostGIS indexes

### Reliability
- 99% uptime for core services
- Graceful degradation when AI services unavailable
- Automatic retry for failed API calls

### Security
- HTTPS for all communication
- CORS restrictions
- Input sanitization
- No persistent storage of symptom data

### Usability
- Mobile-responsive design
- Clear visual hierarchy
- Intuitive navigation
- Minimal user input required
- WhatsApp-style familiar chat interface
